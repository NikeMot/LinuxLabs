# Lab Duration Standard

## Rule

Every LinuxLabs lab must be scoped for **60–90 minutes maximum** of active learner work.

A lab should not become a multi-hour exercise. If the topic requires more than 90 minutes, split it into separate focused labs.

## What Counts Inside the Timebox

The 60–90 minute limit covers the learner's active work:

* reading the scenario and requirements
* planning the implementation
* running commands
* troubleshooting issues
* completing verification checks
* capturing useful command output and observations
* preparing enough evidence for the final write-up

## What Is Outside the Timebox

The following are handled separately after the lab is solved:

* final documentation formatting
* reflection-question capture
* evidence organisation
* repository upload
* final README or lab write-up polishing

## Design Rules

Each lab should have:

* one clear operational scenario
* a small number of focused requirements
* explicit pass/fail evidence
* one controlled break/fix element where practical
* enough cumulative repetition to reinforce previous topics without turning the lab into a long revision session

## Splitting Rule

If a proposed lab includes too many major skills, split it.

Examples:

* user creation, sudo policy, ACLs, and SSH hardening can be one 60–90 minute lab if tightly scoped
* users, storage, networking, systemd, firewalling, monitoring, backups, and automation should be split across multiple labs

## Completion Standard

A lab is successful when the learner can complete the required technical work, capture verification evidence, and explain the outcome within the timebox.
